Swarms Quick Start

Parallel AI Agents for Architects

For developers who know good powder when they see it

What Are Swarms?

Multiple AI agents executing different tasks in parallel

Think of it like this: You've got fresh powder, different types of lines available. You can take them one at a time (sequential), or your whole crew can drop in simultaneously - trees guy, cliffs guy, pillows guy, couloirs guy - each hitting their specialty at the same time.

Same mountain. Same session. Everyone carving their own type of line. Way faster. Way more productive.

The Pattern

  1. Initialize swarm topology (mesh, hierarchical, etc.)
  2. Spawn specialized agents with clear, independent tasks
  3. Agents coordinate via shared memory (no direct communication)
  4. Aggregate results

When To Use (Conditions Report)

Perfect conditions for parallel lines:
  • Fresh powder day: Multi-component projects with different specialties
  • Different line types available: Backend, frontend, infra, tests (all independent)
  • Time-sensitive: Need to maximize output before conditions change
  • You're equipped: MAX plan = unlimited runs
Translation: When you have good powder (complex project) and different lines to hit (independent components), send the whole crew at once.

Skip parallel for:

  • Single feature work (one line to session)
  • Deep debugging (need singular focus)
  • Tight coupling (sequential dependencies)
  • Learning mode (dial in one line first)

Know the difference between a powder day and a park session.

Architecture

Queen Coordinator
    ↓
Shared Memory Layer (MCP)
    ↓
├─ Agent 1: Backend
├─ Agent 2: Frontend
├─ Agent 3: Database
├─ Agent 4: Tests
└─ Agent 5: DevOps

Each agent:
- Reads from shared memory (coordination data)
- Executes independently (no blocking)
- Writes results back (for other agents)
- No direct agent-to-agent communication
- Meets at aggregation point

Common Mistakes

1. Over-parallelization

Not everything should run in parallel. Sequential dependencies exist for a reason.

2. Unclear task boundaries

Each agent needs a well-defined, independent scope. Overlapping responsibilities = conflicts.

3. Ignoring coordination overhead

Small tasks don't benefit from parallelization. Setup cost > time saved.

4. Skipping aggregation

Someone needs to integrate all the pieces. Don't forget the final assembly step.

Rule of thumb: If tasks can't run independently for at least 15-30 minutes each, parallelization overhead isn't worth it.
