Swarms Quick Start Guide

Drop In & Send It

For architects who know when to carve and when to straight-line

The Run Down (Literally)

Swarms = Multiple AI agents working in parallel

Think of it like dropping into a bowl with your crew:

  • Sequential: Everyone waits, then follows the same line one at a time
  • Parallel (Swarms): Everyone drops in together, each carves their own line down the same face, you all meet at the bottom

Same run. Same destination. Each person taking their own line. Way faster. Way more fun.

When To Use (Conditions Report)

Wide Open Bowl (Perfect for swarms):
  • Multi-component projects (backend + frontend + infra)
  • Independent tasks that can run parallel
  • Time-critical builds ("need this by EOD")
  • You have MAX plan (unlimited runs)
Plenty of room for everyone to carve their own line simultaneously

Narrow Chute (Use sequential):

  • Single file changes
  • Deep debugging (need focus)
  • Highly dependent sequential tasks
  • Learning/experimenting (take it slow)

Only room for one skier at a time - wait your turn

Gotchas (Where People Yard Sale)

1. Over-coordination

Don't try to coordinate too tightly. Let agents carve their own lines. Trust the shared memory layer.

2. Unclear task boundaries

Make sure each agent has a clear, independent scope. Overlapping responsibilities = collision.

3. Sequential dependencies

If Task B requires Task A to finish, that's sequential. Don't force it into parallel.

4. Ignoring aggregation

Someone needs to integrate all the pieces. Usually the Queen Coordinator or a dedicated integration agent.

Ski Patrol Advisory: Most wipeouts happen because people try to parallel everything. Some chutes are narrow - only room for one line at a time. Know the difference between a bowl and a couloir.
